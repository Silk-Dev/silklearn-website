# SILKLEARN page-by-page copy audit

Branch: `feature/copy-audit-pages`
Reviewer: Mia
Method used: current built-in copy-review capability (no newly installed copy skill used for this pass)
Date: 2026-03-18

## Executive view

SILKLEARN already has a sharper thesis than most early AI websites, but the copy still has a recurring problem: it speaks in internally accurate product language before it has earned comprehension from a cold visitor.

The strongest raw idea on the site is this:

> SILKLEARN helps teams turn dense internal documentation into a reviewable, usable structure for onboarding, handoffs, and AI context.

The weakest pattern on the site is this:

- too much abstract language too early
- too many near-synonyms for the same thing
- not enough concrete before/after framing
- not enough proof of who this is for, when it is a fit, and what happens after signup
- several pages feel conceptually adjacent rather than conversion-supportive

## Highest-level copy diagnosis

### What is working
- The site has a real thesis, not generic AI sludge.
- The tone is more serious and thoughtful than most startup sites.
- It keeps review/trust in the story, which is a strong differentiator.
- It is clearly aimed at complex internal knowledge, not generic LMS fluff.

### What is hurting conversion
- "knowledge compilation infrastructure," "dependency-aware artifacts," and similar phrases appear before the visitor has a concrete mental model.
- Multiple pages repeat the same thought with different wording instead of advancing the case.
- Some pages explain SILKLEARN in founder-language rather than buyer-language.
- The site underplays the operational outcomes: faster onboarding, fewer handoff errors, less senior-interruption load, better AI context quality.
- Several CTAs are soft or vague relative to the seriousness of the claim.

# 1. Home page (`/`)

## What is there now
The homepage positions SILKLEARN as a structure-first system that turns internal docs into reviewable learning paths, graphs, and context bundles.

## What is weak / generic / unclear / mispositioned
- The headline is too broad: **"Turn internal docs into learning paths."** That undersells the product and risks sounding like a course builder.
- The subheadline is better, but still makes the user do too much decoding.
- The page introduces several output concepts at once: graph, learning path, context bundle, onboarding, rollout, AI. That is intellectually coherent, but conversion-wise it is crowded.
- "SILKLEARN solutions" and "Silklearn in action" are weaker labels than the underlying content.
- The page still does not answer fast enough: **who is this for, what pain does it remove, and what concrete output do I get?**
- The FAQ is one of the clearest sections on the site; that means the core page is not doing enough work up front.

## Proposed improved copy direction
### Positioning direction
Lead with the business problem first, then explain the mechanism.

### Better homepage message shape
- **Headline direction:**
  - Turn dense internal docs into clear onboarding and AI-ready knowledge.
  - Or: Make complex internal knowledge usable without rebuilding everything by hand.
- **Subheadline direction:**
  - SILKLEARN maps prerequisite order, hidden assumptions, and source-backed structure so leaders can review what teams and AI will use before mistakes spread.
- **Core framing to emphasize:**
  - You already have the knowledge.
  - The problem is that it is trapped in documents and expert memory.
  - SILKLEARN turns that into a reviewed structure teams can actually use.

### Section-level direction
- "SILKLEARN solutions" → **Where SILKLEARN helps first**
- "Silklearn in action" → **What teams do with it**
- "Built for teams where missing context creates expensive mistakes" is strong; keep that logic and make it earlier.
- CTA section should be more explicit about what happens next: fit review, pilot conversation, early-access workflow.

## Rewrite priority
**Rewrite first.**
This page sets comprehension for everything else, and right now it is thoughtful but slower than it should be.

- "knowledge compilation infrastructure" is probably true, but too internal/jargon-heavy as the lead phrase.

### Positioning direction

### Better copy direction

- Then explain how it does that.

The input → compilation → outputs section is good and should remain, but the output column should emphasize business outcomes more directly.

## Rewrite priority
High. This is likely the second page to rewrite after the homepage.

# 3. How it works page (`/how-it-works`)

## What is there now
A five-pass explanation of the SILKLEARN pipeline.

## What is weak / generic / unclear / mispositioned
- "A 5-pass compiler" is memorable, but may be too technical for many buyers unless the audience is extremely technical.
- The page is clear for someone who likes systems language, but it still foregrounds mechanism over payoff.
- Some terms are precise but not intuitive: "segment coherent units," "reconcile overlaps and contradictions."

## Proposed improved copy direction
### Positioning direction
Keep the structured-process framing, but anchor each step in why the user should care.

### Better copy direction
For each step, use a **what happens / why it matters** pattern:
- Read the real source material → so the output stays grounded in the docs your team already trusts.
- Break knowledge into usable units → so onboarding and AI context are not built from random chunks.
- Map prerequisites and dependencies → so teams learn in the right order.
- Reconcile overlap and contradictions → so leaders catch conflicts before rollout.
- Review before release → so the final output is trusted, not blindly generated.

### Specific messaging direction
Reduce "compiler" language slightly unless SILKLEARN is deliberately targeting technical leaders first.

## Rewrite priority
Medium-high. Useful page, but not the first page that needs surgery.

# Cross-page patterns to fix everywhere

## 1. Reduce internal-language density
Words and phrases to use more selectively:
- knowledge compilation infrastructure
- dependency-aware artifacts
- defensible order
- context bundles
- structure-first knowledge operations

These can remain in supporting copy, but should not carry first-contact comprehension.

## 2. Pick one dominant promise
Right now SILKLEARN is trying to be understood as:
- onboarding infrastructure
- knowledge graph system
- review layer
- learning-path engine
- AI context system

The better hierarchy is:
1. turn dense internal docs into usable, reviewed structure
2. that structure powers onboarding, handoffs, and AI context
3. leaders review before downstream use

## 3. Make outcomes more concrete
More of these:
- reduce ramp time
- reduce repeated expert interruption
- reduce wrong-sequence learning
- improve handoff consistency
- improve AI context quality

## 4. Clarify the buying motion
The site should explain:
- who should join the waitlist
- what makes a team a fit
- what happens after signup
- whether this is a pilot, beta, or early-access workflow

## 5. Separate thesis pages from conversion pages
Guides/blog can hold more abstract language.
Homepage/product/waitlist should be clearer, faster, and more concrete.

# Which page should be rewritten first?

## Recommendation
**Rewrite the homepage first.**

## Why
- It currently does the most conceptual heavy lifting.
- It sets the frame for every other page.
- If the homepage becomes clearer and more concrete, the rest of the site's copy burden drops.
- The current homepage is intelligent but still makes visitors translate too much too early.

## Immediate rewrite order after that
1. Home page
2. Waitlist page
3. Product page
4. How it works page
5. Leader review feature page
6. Engineering onboarding use-case page

SILKLEARN helps teams turn dense internal documentation into reviewed, usable structure for onboarding, handoffs, and AI context.

## Supporting differentiators
